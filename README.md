# Practices-for-Secure-Software-Report-for-Artemis-Financials
Refactored Spring Boot application for Artemis Financial demonstrating secure coding practices: HTTPS with self-signed certificate, SHA-256 checksum verification, and OWASP Dependency-Check integration. CS 305 Project Two (SNHU).

## The Client and Their Needs
Artemis Financial is a fictional financial services company that provides investment, retirement, and insurance consulting to individual clients. Because they handle sensitive personal and financial data for their customers, they need to operate in a regulated environment where even a single breach can result in significant legal liability, regulatory penalties, and reputational damage. The company engaged me to modernize the security posture of their Spring Boot web application. Specifically, they needed me to enable encrypted HTTPS communication between clients and the server, add cryptographic integrity verification so data tampering could be detected, and establish a repeatable process for identifying vulnerabilities in the third-party libraries their application depends on.
## What I Did Well
Where I felt strongest was in systematically working through the vulnerability assessment rather than jumping straight to fixes. Before writing any code, I looked at what the starter application was actually doing, what layers of the stack were exposed, and what security controls were missing. I also took the time to justify my cryptographic choices — selecting SHA-256 not because it was the first option I found, but because it met specific industry criteria (FIPS 140-2 approval, collision resistance, NIST recommendation through 2030 and beyond). Coding securely matters because software is increasingly the primary interface between a business and its customers, and every line of insecure code is a potential entry point for an attacker. For a financial company, software security isn't just a technical concern — it directly protects the company's regulatory standing, customer trust, and long-term business viability. A single publicly disclosed breach can cost millions in direct remediation, but the reputational damage can cost far more in customer churn and lost future revenue.
## What Was Challenging
The most challenging part was interpreting the OWASP Dependency-Check report. The initial scan returned 160 vulnerabilities across 13 dependencies, and the instinct is to panic and try to fix everything at once. Learning to triage the findings — identifying which were critical versus informational, which applied to my actual usage versus which were theoretical for configurations I wasn't using, and which were on CISA's Known Exploited Vulnerabilities list versus being published but not actively exploited — was a skill I didn't fully have before this project. The most helpful part, conversely, was how concrete the feedback loop became once the dependency-check plugin was integrated into Maven. Being able to run one command and get an authoritative report of every known vulnerability in my dependency tree turned security from an abstract concept into something measurable.
## Increasing Layers of Security
I added security at three distinct layers rather than relying on a single control. At the transport layer, I generated a self-signed X.509 certificate with a 2048-bit RSA key and configured Spring Boot to enforce TLS on all incoming connections. At the application layer, I built a SHA-256 checksum endpoint so that data integrity can be verified by downstream systems. At the dependency layer, I integrated OWASP Dependency-Check into the Maven build pipeline so vulnerability scanning happens automatically on every compile rather than as a one-time event. In future work, I would pair OWASP Dependency-Check with additional tools like SonarQube or Snyk for source-code-level static analysis, and I would consult the OWASP Top 10 and CWE Top 25 lists when deciding which mitigation techniques to prioritize. These frameworks provide a principled way to rank threats by real-world impact rather than guessing.
## Functional Testing and Verifying No New Vulnerabilities
I verified functionality by manually testing every endpoint after each refactoring step — first confirming the application still built cleanly, then confirming HTTPS actually served traffic at https://localhost:8443, then confirming the new /hash endpoint returned a valid 64-character SHA-256 digest. To confirm I hadn't introduced new vulnerabilities, I re-ran the OWASP Dependency-Check scan after all my refactoring was complete and compared the results. The key insight here is that scanning once isn't enough — the real value of a static analysis tool comes from running it repeatedly throughout the development lifecycle so you can detect any regression the moment it's introduced.
## Useful Resources, Tools, and Practices
Several things from this project will carry forward into future work. Java Keytool for generating and managing keystores is a tool I'll use any time I need to stand up TLS for a development environment. The OWASP Dependency-Check Maven plugin is something I'd add to any Java project going forward as a matter of routine, not just when security is the explicit focus. The broader practice of shifting security left — catching problems at build time instead of in production — is a DevSecOps principle I want to apply to every project I work on. On the documentation side, the habit of justifying each security decision with reference to an authoritative source (NIST, OWASP, CISA) rather than just "this is what everyone uses" is something I want to carry into professional practice.
## What I'd Show an Employer
This project is a good portfolio piece because it demonstrates end-to-end ownership of a security refactor rather than isolated knowledge of individual security concepts. I'd show an employer the refactored code alongside the Practices for Secure Software Report, because together they show I can not only implement security controls but also communicate clearly about tradeoffs and justify my choices to a non-technical stakeholder. Specifically, the SHA-256 implementation demonstrates I can work with Java's cryptography APIs correctly, the TLS configuration demonstrates I understand certificate management fundamentals, and the OWASP Dependency-Check integration demonstrates I think about the software supply chain rather than just the code I personally write. That last piece — caring about the security of the libraries my code depends on — is increasingly what separates competent developers from junior ones in the current landscape.
